## Blocker
### 1. Dashboard card permissions now depend on a field the list API does not return
Location: `frontend/src/components/DrawingCard.tsx` in the hunk changing `draggable`, rename/delete/move gating, and the collection chip label; paired with `backend/src/routes/dashboard/drawings.ts` `GET /drawings` summary response.

Issue description: The PR replaces the old `isShared` checks with `drawing.accessLevel === 'owner'` / `'edit'`, but `GET /drawings` still returns plain summaries without any `accessLevel`. Outside `/drawings/shared`, that means `drawing.accessLevel` is `undefined` for normal owner-owned rows. The result is a broad regression: owned drawings stop being draggable, owner-only menu actions disappear, and the collection pill falls back to `Shared` because `drawing.accessLevel !== 'owner'` is true for every normal row. The feature mostly works in the editor because `GET /drawings/:id` does return `accessLevel`, but the dashboard list is now lying about ownership.

Proposed fix: Either keep the old `isShared`-based gating for the existing owner-only list, or update `GET /drawings` to emit a correct per-row `accessLevel` for every returned drawing. For shared collections that must be computed row-by-row, not inferred from the current route.

## Major
### 2. Bulk move/delete still lets collaborators act on drawings they do not own
Location: `frontend/src/pages/Dashboard.tsx` in `handleBulkDeleteClick`, `executeBulkMoveToTrash`, `handleBulkMove`, `handleDrop`, and the bulk-action button disable logic.

Issue description: The PR summary says only drawing owners can delete, move, or duplicate their own drawings. The single-card menu was updated toward that model, but the bulk flows still operate on `selectedIds` with no ownership filtering. In a shared collection, a collaborator can select someone else’s drawings and trigger bulk delete or bulk move. The UI optimistically removes or relocates those cards before the backend rejects some requests with `403`/`404`, so the page will flicker into a false state and then recover only after a refresh. That is an authorization mismatch and a bad failure mode.

Proposed fix: Before enabling or executing bulk actions, restrict the candidate set to drawings with owner access. If mixed selections are allowed, either filter out foreign drawings and explain it in the UI or disable the action entirely with a clear message.

## Minor
### 3. Bulk duplicate is disabled by collection ownership, not drawing ownership
Location: `frontend/src/pages/Dashboard.tsx` in the `disableBulkActions` calculation and the "Duplicate Selected" button.

Issue description: `disableBulkActions` is set for any shared collection where the viewer is not the collection owner. That blocks bulk duplication even for drawings the current user owns inside someone else’s shared editable collection. This conflicts with the PR’s own stated rule that drawing owners can duplicate their own drawings. It also creates an inconsistent model where ownership is per-drawing in the backend and per-collection in the dashboard toolbar.

Proposed fix: Base bulk duplication eligibility on the selected drawings’ ownership/access level, not on `currentCollection.isOwner`.
